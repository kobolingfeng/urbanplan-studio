# Case Validation Protocol

本文件用于把 UrbanPlan Studio 的软件演示转化为硕士论文中可记录、可复核、可答辩的案例验证流程。

## 1. 验证目标

案例验证不证明系统已经达到商业规划平台水平，而是验证原型是否能在城市更新早期方案推演中形成稳定的研究证据链：

- 能否用 UPF 表达地块、道路、设施、出入口、开放空间、控制线和多方案情景。
- 能否把规则问题、指标评分、数据质量和证据来源关联起来。
- 能否对不同方案给出可解释的比较结果。
- 能否通过权重敏感性说明方案推荐是否受价值偏好影响。
- 能否导出可复现的 Markdown 和 UPF 材料，支撑论文方法、实验和结果章节。

## 2. 案例材料

建议每个案例至少保留以下材料：

| 材料 | 来源 | 用途 |
|---|---|---|
| UPF 项目文件 | 软件保存 | 复现实验数据 |
| 规划诊断报告 | `报告` | 描述当前方案指标、问题和建议 |
| 方案综合评估 | `评估` | 解释评分维度、地块优先级和风险 |
| 方案决策矩阵 | `对比` | 记录多方案比较结果 |
| 权重敏感性分析 | `敏感` | 说明推荐结果稳健性 |
| 数据质量诊断 | `质检` | 暴露证据覆盖率、规则目录和导入审计 |
| 案例验证包 | `验证` | 汇总研究问题、证据链、实验表和 CSV 附录 |

## 3. 记录步骤

1. 选择一个真实或课程设计片区，确定研究边界、规划类型和更新目标。
2. 将地块、道路、设施、出入口、开放空间和约束控制线录入 UPF。
3. 为每个对象补充证据来源，至少记录资料名称、年份或获取方式。
4. 建立不少于两个情景方案，例如现状基准、更新增强、公共服务优先或保护修复优先。
5. 运行 `检查`，记录硬性错误、警告和建议。
6. 分别导出 `评估`、`对比`、`敏感`、`质检` 和 `验证` 报告。
7. 邀请导师、同学或规划相关人员复核规则结果和方案排序。
8. 在论文中区分“系统自动生成结果”和“专家复核意见”。

## 4. 评价指标

| 指标 | 计算方式 | 论文位置 |
|---|---|---|
| 对象完整性 | UPF 对象数量、类型覆盖、缺失关系 | 案例数据构建 |
| 规则识别能力 | 错误、警告、提示数量及专家复核 | 规则验证 |
| 方案可比性 | 多方案评分、人口、建面、风险计数 | 方案比较 |
| 稳健性 | 四类权重模型下的第一名一致性 | 敏感性分析 |
| 数据可信度 | 证据覆盖率、质量分、导入审计 | 数据质量 |
| 可用性 | 任务完成率、用时、专家评分 | 用户/专家评价 |

## 5. 结论边界

推荐采用保守表述：

- 可以说系统提升了案例材料整理、规则解释和方案比较的一致性。
- 可以说系统为早期方案推演提供了可追溯的辅助判断。
- 不应说系统替代法定控规审查、交通影响评价、消防审查或专家判断。
- 当证据覆盖率不足时，应把结果定位为原型演示或形成性验证。
