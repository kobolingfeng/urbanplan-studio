# UrbanPlan Studio 评价方法

## 1. 评价目的

本文件定义 UrbanPlan Studio 作为城市更新/城市规划硕士毕业设计原型的评价方法。评价目标不是证明原型已经达到商业规划平台水平，而是验证它是否能在毕业设计和早期方案推演中提供可复现、可解释、可比较的研究支撑。

评价围绕三个核心主张展开：

1. 语义化建模可以降低图形、指标、规则和证据分散造成的错误。
2. 情景比较与规则检查可以提高城市更新方案分析的效率和一致性。
3. 数据质量与证据链显式化可以增强毕业设计答辩中的可解释性。

## 2. 评价原则

评价应遵循以下原则：

1. 可复现：评价数据、任务、指标公式和评分规则应被记录，避免只展示界面截图。
2. 可解释：所有规则结果都应说明对象、阈值、实际值、严重等级和依据类型。
3. 分层评价：区分功能正确性、规划合理性、用户可用性和工程可靠性。
4. 保守结论：原型规则不能被表述为法定审查结论，实验结果应限定在样本和案例范围内。
5. 证据优先：用户反馈、专家评分、日志记录和系统导出报告应共同构成评价材料。

## 3. 评价对象

评价对象包括：

1. UPF 语义规划格式及导入导出流程。
2. 地块、道路、设施、出入口、开敞空间和约束覆盖区的编辑与展示。
3. 规则检查结果，包括强度控制、设施容量、设施覆盖、出入口风险和保护覆盖风险。
4. 情景比较结果，包括容量、公共服务、风险数量和数据质量变化。
5. 数据质量报告，包括缺失证据、引用完整性、情景完整性和原型规则提示。
6. Markdown 报告输出与答辩材料可用性。
7. 工程验证流程，包括类型检查、几何测试、UPF 测试、规则测试和打包验证。

## 4. 指标体系

### 4.1 功能完整性

| 指标 | 定义 | 记录方式 | 合格标准 |
| --- | --- | --- | --- |
| 核心流程完成率 | 评价任务中成功完成的核心步骤比例 | 手工验收表 | 不低于 90% |
| 对象编辑成功率 | 新建、选择、修改、删除对象的成功次数占比 | 任务日志 | 不低于 90% |
| 导入导出成功率 | UPF 保存、载入、导出和错误提示成功次数占比 | 测试记录 | 不低于 95% |
| 报告生成完整度 | 报告是否包含情景、规则、风险、建议和数据质量 | 报告核查表 | 必须包含全部核心项 |

### 4.2 规则检查正确性

| 指标 | 定义 | 记录方式 | 合格标准 |
| --- | --- | --- | --- |
| 规则命中准确率 | 系统命中的违规项中经人工复核正确的比例 | 规则夹具和专家复核 | 不低于 85% |
| 规则召回率 | 人工标注违规项中被系统发现的比例 | 规则夹具和专家复核 | 不低于 80% |
| 严重等级一致率 | 系统严重等级与专家评分一致或相邻的比例 | 专家评分表 | 不低于 85% |
| 结果可解释率 | 检查结果包含对象、原因、阈值或建议的比例 | 结果抽样核查 | 不低于 90% |

可使用如下混淆矩阵记录规则验证：

| 人工判断 / 系统判断 | 系统提示风险 | 系统未提示风险 |
| --- | --- | --- |
| 实际有风险 | TP | FN |
| 实际无风险 | FP | TN |

计算公式：

```text
准确率 = TP / (TP + FP)
召回率 = TP / (TP + FN)
F1 = 2 * 准确率 * 召回率 / (准确率 + 召回率)
```

### 4.3 情景分析有效性

| 指标 | 定义 | 记录方式 | 合格标准 |
| --- | --- | --- | --- |
| 情景指标一致性 | 同一情景指标在系统表格、报告和人工复核中的一致程度 | 抽样核算 | 误差不超过 2% 或原因可解释 |
| 方案差异可读性 | 用户能否根据比较表说出主要优劣差异 | 任务问卷 | 平均分不低于 4/5 |
| 风险变化识别率 | 用户能否识别不同情景下新增、减少或转移的风险 | 任务答题 | 正确率不低于 80% |
| 决策支撑度 | 专家认为输出能否支撑方案讨论 | 专家 Likert 评分 | 平均分不低于 4/5 |

### 4.4 数据质量与证据链

| 指标 | 定义 | 记录方式 | 合格标准 |
| --- | --- | --- | --- |
| 必填字段完整率 | UPF 对象必填字段完整数量占比 | 数据质量报告 | 不低于 95% |
| 证据标注率 | 关键对象或规则结果具有证据说明的比例 | 数据抽样 | 不低于 80% |
| 引用完整率 | 出入口等对象引用有效地块和道路的比例 | 数据质量报告 | 100% |
| 原型规则标识率 | 启发式或原型规则被明确标识的比例 | 规则结果核查 | 100% |

### 4.5 可用性与效率

| 指标 | 定义 | 记录方式 | 合格标准 |
| --- | --- | --- | --- |
| 任务完成时间 | 完成指定方案检查与比较任务所需时间 | 屏幕记录或计时 | 相比传统流程减少 20% 以上 |
| 用户错误次数 | 操作中出现误选、漏填、错算、忘记保存等错误次数 | 观察记录 | 相比传统流程减少 20% 以上 |
| SUS 可用性评分 | System Usability Scale 量表得分 | 问卷 | 不低于 70 |
| 学习负担 | 用户完成首轮任务后的主观难度评分 | 5 分量表 | 平均不高于 3/5 |

### 4.6 工程可靠性

| 指标 | 定义 | 记录方式 | 合格标准 |
| --- | --- | --- | --- |
| 自动验证通过率 | 类型检查、几何、UPF、规则、规则夹具、评价、GeoJSON、CSV、Markdown、证据、范围、dev-utils、鲁棒性、静态构建和发布检查通过比例 | `bun run verify` 记录 | 100% |
| 发布包可用性 | 打包产物是否可启动并通过冒烟测试 | 发布验证记录 | 100% |
| 错误恢复能力 | 导入无效 UPF、删除被引用对象等异常操作是否有明确提示 | 手工验收 | 核心异常均可处理 |
| 数据往返一致性 | 保存再载入后对象数量和关键属性是否一致 | 测试记录 | 关键字段一致 |

### 4.7 当前原型综合评分模型

当前代码中的 `planning-evaluation.ts` 已实现一套可解释的原型评分模型。它不是法定审查结论，而是用于毕业设计案例比较和答辩说明的决策支持指标。

| 维度 | 默认权重 | 当前计算含义 | 后续校准方式 |
| --- | ---: | --- | --- |
| 控规符合性 | 24% | 按错误、警告和提示扣分 | 专家复核严重等级 |
| 公共服务 | 22% | 公服建面比例、幼儿园、养老、卫生容量与覆盖 | 依据地方完整社区标准校准 |
| 交通组织 | 16% | 路网密度、地块到道路距离、出入口风险 | 引入路网步行距离和交评指标 |
| 生态与开放空间 | 16% | 绿地率达标度与人均开放空间 | 加入蓝绿网络和热环境指标 |
| 更新价值 | 12% | 强度适配、更新方式、公服增量 | 专家 AHP 或熵权法校准 |
| 证据可信度 | 10% | 对象证据覆盖、规则依据数量、原型规则扣分 | 结构化 EvidenceSource |

论文中可把该模型表述为“原型阶段的透明加权评分”。若使用 AHP、熵权法或专家打分，应保留本表作为初始模型，并在案例验证章节说明权重变化对推荐方案的敏感性。

### 4.8 权重敏感性分析

系统内置四组权重画像：均衡模型、公共服务优先、保护与生态优先、实施风险优先。每组权重都重新计算全部方案得分和排序，用于检验推荐方案是否依赖单一价值偏好。

论文中建议记录：

1. 不同权重模型下的第一名方案是否一致。
2. 当前方案在各模型下的得分变化。
3. 第一名与最后一名的分差范围。
4. 推荐结果变化时，对应的是公共服务、生态保护还是实施风险等价值取向差异。

若多个模型推荐同一方案，可表述为“案例范围内推荐结果具有一定稳健性”。若推荐结果随权重变化明显，应表述为“方案选择对价值偏好敏感”，并解释不同偏好下的规划含义。

## 5. 实验设计

### 5.1 被试与角色

建议招募 8 到 15 名被试，分为两类：

1. 学生用户：城市规划、建筑学或城市设计方向研究生，熟悉 CAD/表格但不一定熟悉本原型。
2. 专家评审：具有城市更新、控规、公共服务设施或历史保护经验的教师、规划师或高年级研究生。

若样本不足，可将用户实验定位为形成性评价，并把专家复核与案例验证作为主要证据。

### 5.2 对照流程

建议设置两种流程：

1. 传统流程：使用 CAD 图纸、Excel 表格和人工规范说明完成指标核查与情景比较。
2. UrbanPlan Studio 流程：使用原型完成同样任务，导出规则结果、比较表和报告。

为减少学习效应，可采用交叉设计：一半被试先使用传统流程再使用原型，另一半顺序相反。两个流程使用相同案例和任务说明。

### 5.3 任务设置

任务应控制在 30 到 45 分钟内，覆盖核心能力：

1. 打开案例项目并识别主要对象。
2. 修改某地块的容积率、绿地率和更新模式。
3. 运行规则检查并解释至少三条风险提示。
4. 比较 S0、S1、S2 三个情景的容量、设施和风险变化。
5. 发现一个数据质量问题并说明其影响。
6. 导出报告，并判断其是否能作为答辩材料的一部分。

### 5.4 数据采集

应采集以下数据：

1. 任务完成时间。
2. 任务成功或失败状态。
3. 操作错误次数。
4. 规则结果人工复核表。
5. 情景指标人工核算表。
6. 用户问卷，包括 SUS 和 5 分量表。
7. 专家评分，包括规划合理性、结果可解释性和答辩支撑度。
8. 系统导出的 UPF、比较表、数据质量报告和 Markdown 报告。

质检报告应同时保留 UPF 结构校验、导入审计和规则目录元数据。这样论文中可以把“数据是否完整”和“规则是否可追溯”拆开论证，而不是只展示最终评分。

## 6. 案例验证方法

案例验证应选择一个具有真实规划语境的城市更新片区。数据可以来自公开规划资料、课程调研、毕业设计自绘底图或教师提供材料，但必须记录来源和加工过程。

### 6.1 案例数据要求

案例至少应包含：

1. 3 个以上地块。
2. 2 条以上道路或道路段。
3. 2 类以上公共服务设施。
4. 1 个以上开敞空间。
5. 1 个以上约束覆盖区，例如历史保护、生态控制、退界或风险区。
6. 3 个情景方案：现状基准、微更新、综合更新。

当前仓库提供 `examples/luohu-case-v1.upf` 作为案例验证起点。该样例包含 3 个地块、3 条道路、3 类公共服务设施、1 个开放空间、1 个风貌约束区和 3 个情景方案；配套数据来源登记与复核模板位于 `docs/cases/luohu/`。若用于正式论文，应将教学示意数据逐步替换为公开规划资料、GIS 数据、现场调研或导师提供资料。

### 6.2 案例验证步骤

1. 数据整理：将案例资料转化为 UPF 对象，并记录来源。
2. 现状校核：运行 S0 检查，识别现状问题和数据质量缺口。
3. 情景建模：设置 S1 和 S2 的地块强度、绿地率、公共服务面积和更新模式。
4. 规则检查：分别运行各情景规则检查，记录风险数量和严重等级。
5. 情景比较：输出容量、服务、风险和质量变化表。
6. 专家复核：请专家对风险结果和建议进行正确性与合理性评分。
7. 报告审查：检查系统报告是否能支撑论文图表、答辩问答和设计说明。

### 6.3 案例输出模板

案例结果建议以如下结构呈现：

| 项目 | S0 现状基准 | S1 微更新 | S2 综合更新 | 说明 |
| --- | --- | --- | --- | --- |
| 总建筑量 |  |  |  | 记录计算口径 |
| 平均容积率 |  |  |  | 说明地块范围 |
| 绿地率风险数 |  |  |  | 对应规则结果 |
| 公服容量缺口 |  |  |  | 按设施类型分解 |
| 服务覆盖缺口 |  |  |  | 当前为直线距离 |
| 保护风险数 |  |  |  | 区分相交与拆建 |
| 数据质量分 |  |  |  | 记录缺失证据 |

## 7. 专家评分表

专家可按 1 到 5 分评分：

| 维度 | 1 分 | 3 分 | 5 分 |
| --- | --- | --- | --- |
| 规划对象表达 | 对象含义混乱 | 基本能表达主要对象 | 对象、属性和情景关系清晰 |
| 规则结果正确性 | 多数结果不可信 | 主要问题可识别 | 结果准确且便于追溯 |
| 建议可操作性 | 建议空泛 | 部分建议可参考 | 建议能指导方案修改 |
| 情景比较价值 | 难以支持判断 | 能说明部分差异 | 能清楚支撑方案取舍 |
| 证据链完整性 | 依据不明 | 主要依据可见 | 规则、数据和假设区分清楚 |
| 答辩支撑度 | 难以引用 | 可作为辅助材料 | 可直接支撑关键论证 |

同时记录开放式意见：

1. 哪些规则结果最有帮助？
2. 哪些提示可能误导用户？
3. 还需要补充哪些城市更新评价维度？
4. 报告中哪些内容适合进入论文或答辩展示？

## 8. 数据分析方法

定量分析：

1. 对任务时间、错误次数和问卷分数计算均值、中位数和标准差。
2. 对传统流程与原型流程比较任务时间和错误次数，可使用配对样本统计或描述性比较。
3. 对规则验证计算准确率、召回率和 F1。
4. 对专家评分计算平均分，并标注分歧较大的维度。

定性分析：

1. 对用户访谈和专家意见进行主题归纳。
2. 将反馈归类为数据模型、规则逻辑、交互流程、报告表达和研究边界。
3. 选取典型问题截图或报告片段，说明原型如何帮助或限制了方案论证。

## 9. 有效性威胁

论文中应主动讨论以下威胁：

1. 样本规模有限：毕业设计阶段用户数量不足，结果更适合作为原型评价。
2. 案例代表性有限：单一片区不能覆盖所有城市更新类型。
3. 规则来源有限：部分规则为启发式原型规则，不能等同于正式规范审查。
4. 学习效应：被试对第二种流程更熟悉，可能影响效率比较。
5. 数据质量依赖输入：系统能发现部分缺口，但不能保证原始资料真实完整。
6. 几何简化：直线服务半径和 demo 单位系统会影响设施覆盖判断。
7. 专家主观性：专家评分受经验背景影响，应记录评审人背景并保留开放意见。

## 10. 评价结论写法

建议在论文中采用保守且可验证的结论表达：

1. 可以说：原型在案例范围内提高了规则检查和情景比较的可复现性。
2. 可以说：系统报告能够帮助解释方案风险、数据缺口和指标差异。
3. 可以说：UPF 语义格式适合作为毕业设计阶段的轻量规划数据中间层。
4. 不应说：系统已经实现正式规划审批、完整法定合规审查或真实数字孪生平台能力。
5. 不应说：小样本实验能够证明系统对所有城市更新项目普遍有效。

最终评价结论应同时包含：

1. 哪些研究主张获得支持。
2. 哪些指标未达到预期。
3. 哪些失败或误判暴露了系统边界。
4. 下一步需要补充的数据、规则、算法和工程工作。
