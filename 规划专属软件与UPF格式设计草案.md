# 规划专属软件与 UPF 格式设计草案

设计日期：2026-05-23  
定位：为“非 CAD 的城市规划专属软件”建立第一版产品、数据格式和规则引擎设计。  
关联文件：[控规与智能规划助手研究笔记.md](./控规与智能规划助手研究笔记.md)、[硕士毕业设计总体方案.md](./硕士毕业设计总体方案.md)、[UPF 0.1 与 MVP 闭环设计.md](./UPF%200.1%20与%20MVP%20闭环设计.md)

> 核心判断：我们可以完全不把 CAD 当作主平台。CAD 只是一种传统图纸输入/输出格式，而不是规划软件的内部语言。真正的核心应是“规划对象模型 + 规则库 + 证据系统 + 方案推演引擎”。

## 1. 为什么不必从 CAD 开始

CAD 的优势是精确绘制、图层管理和工程出图，但它天然不理解规划语义。

在 CAD 里，一块住宅用地可能只是一个闭合多段线加一段文字；一个城市道路可能只是两条线；一个出入口可能只是一个符号；一个控规指标可能只是表格里的字符。软件无法天然知道：

- 这个多边形是不是一个合法地块
- 它属于哪个街坊、哪个控规单元、哪个行政区
- 它的用地性质是否允许住宅、商业或混合功能
- 容积率、建筑密度、绿地率是否越界
- 它服务多少人口、需要多少公共服务设施
- 道路出入口是否合理
- 消防、日照、交通、市政、历史保护等约束是否冲突
- 这个判断依据来自哪条规范、哪个城市、哪个版本

所以如果继续沿着 CAD 做，只是在“线和面”上叠加智能；如果新建规划专属软件，就可以直接让软件理解“地块、道路、设施、人口、规则、证据、方案”。

## 2. 软件一句话定义

暂定产品名：UrbanPlan Studio，中文可叫“智规工坊”或“城市规划推演平台”。

一句话定义：

> 一个面向城市更新、控规编制、片区规划和方案审查的语义化规划软件，让设计者不只是画图，而是在可计算、可追溯、可审查的规划对象系统里生成和比较方案。

它不是：

- 不是传统 CAD 替代品
- 不是单纯 GIS 地图
- 不是只会聊天的 AI 助手
- 不是只打分的城市更新潜力平台

它更像：

- 规划版 Figma：多人协作、方案版本、对象化编辑
- 规划版 BIM：每个空间对象有属性、关系和规则
- 规划版 GIS：所有对象都有真实空间位置和外部证据
- 规划版 IDE：规则检查、错误提示、自动建议、版本管理

## 3. 核心设计原则

### 3.1 规划对象优先

内部不以图层、线段、块参照为核心，而以规划对象为核心：

- Project 项目
- PlanningUnit 规划单元
- Block 街坊
- Parcel 地块
- RoadSegment 道路段
- Intersection 交叉口
- Entrance 出入口
- BuildingMass 建筑体量
- Facility 公共服务设施
- OpenSpace 公共开放空间
- GreenSpace 绿地
- Utility 市政设施
- ControlLine 控制线
- ConstraintOverlay 约束覆盖层
- Scenario 方案
- RuleSet 规则集
- EvidenceSource 证据来源

几何只是对象的一个属性，不是对象本身。

### 3.2 规则可追溯

每一个检查结果都必须能回答：

- 违反了什么规则
- 规则来自哪个标准或地方规定
- 使用的是哪个版本
- 规则是强制性、推荐性，还是软件推断
- 输入数据来自哪里
- 计算过程是什么
- 如果数据不确定，不确定性在哪里

这会让软件比“专家拍脑袋”更有论文价值和产品可信度。

### 3.3 方案可比较

规划不是求唯一答案，而是在多约束下比较方案。

软件应支持：

- 现状方案 Baseline
- 方案 A：保守更新
- 方案 B：强度提升
- 方案 C：公共服务优先
- 方案 D：交通优化优先

每个方案可以比较：

- 建筑量
- 居住人口
- 就业人口
- 公共设施缺口
- 道路交通压力
- 绿地和开放空间
- 规范风险
- 更新成本和实施难度
- 对历史文化、生态和社区的影响

### 3.4 证据和推断分开

软件必须区分“原始证据”和“模型推断”。

例如：

- 证据：地块面积来自测绘数据
- 证据：POI 来自高德或 OSM
- 证据：建筑轮廓来自公开建筑物数据
- 推断：建筑层数由高度估算
- 推断：居住人口由住宅建筑面积估算
- 推断：交通压力由开发量和道路等级估算

如果不区分证据和推断，软件会显得不专业；如果区分清楚，即使数据不完美，也能形成可信的辅助决策系统。

## 4. 新格式：UPF

暂定格式名：UPF，Urban Planning Format，城市规划语义格式。

### 4.1 UPF 要解决什么

UPF 不是为了保存图纸，而是为了保存“可计算的规划方案”。

它需要保存：

- 空间几何
- 规划对象
- 对象属性
- 对象关系
- 指标计算
- 规则检查
- 外部证据
- 多方案版本
- 来源和时间
- 不确定性
- 导出成果

### 4.2 文件形态

第一阶段可以让 `.upf` 本质上是一个压缩包，内部由 JSON、GeoJSON、规则文件和资源文件组成。

建议结构：

```text
project.upf
  manifest.json
  project.json
  jurisdiction.json
  objects/
    planning_units.geojson
    blocks.geojson
    parcels.geojson
    roads.geojson
    buildings.geojson
    facilities.geojson
    open_spaces.geojson
    control_lines.geojson
  scenarios/
    baseline.json
    scenario_a.json
    scenario_b.json
  rulesets/
    national_2026.json
    shenzhen_2025.json
    shanghai_2020.json
  evidence/
    sources.json
    poi.geojson
    transit.geojson
    imagery_index.json
  checks/
    baseline_check.json
    scenario_a_check.json
  reports/
    scenario_a_summary.md
```

第一版用 JSON/GeoJSON 是因为好开发、好调试、好写论文。未来如果数据量变大，可以换成 GeoPackage、FlatGeobuf、Parquet 或数据库。

### 4.3 manifest.json

`manifest.json` 记录文件身份和版本。

```json
{
  "format": "UPF",
  "formatVersion": "0.1.0",
  "createdAt": "2026-05-23T00:00:00+08:00",
  "updatedAt": "2026-05-23T00:00:00+08:00",
  "software": {
    "name": "UrbanPlan Studio",
    "version": "0.1.0"
  },
  "coordinateReferenceSystem": "EPSG:4490",
  "defaultScenarioId": "scenario_a",
  "jurisdictionId": "CN-GD-SZ"
}
```

### 4.4 project.json

`project.json` 保存项目基本信息。

```json
{
  "id": "project_luohu_update_001",
  "name": "罗湖某片区城市更新推演",
  "type": "urban_renewal",
  "scale": "block_or_district",
  "studyBoundaryObjectId": "unit_001",
  "planningHorizon": {
    "baseYear": 2026,
    "targetYear": 2035
  },
  "goals": [
    "补齐公共服务设施",
    "优化慢行和公交可达性",
    "控制开发强度风险",
    "提升公共开放空间"
  ]
}
```

## 5. 核心对象模型

### 5.1 Project 项目

项目是所有数据、规则和方案的容器。

关键字段：

- 项目名称
- 类型：控规、城市更新、片区规划、住区规划、专项规划
- 规划范围
- 所在城市和行政区
- 基准年和目标年
- 适用规则集
- 方案列表

### 5.2 PlanningUnit 规划单元

规划单元是控规或详细规划的基本组织范围。

关键字段：

- 单元编号
- 面积
- 所属行政区
- 上位规划约束
- 主导功能
- 发展目标
- 总人口控制
- 总建设量控制
- 公共服务设施要求
- 生态、历史、交通、市政约束

### 5.3 Block 街坊

街坊是道路围合或规划管理形成的中间单元。

关键字段：

- 街坊编号
- 所属规划单元
- 面积
- 主导功能
- 街坊人口
- 街坊建筑量
- 街坊公共服务缺口
- 与周边道路和设施的关系

### 5.4 Parcel 地块

地块是最重要的规划对象。

关键字段：

- 地块编号
- 用地分类代码
- 用地兼容性
- 面积
- 容积率
- 建筑密度
- 绿地率
- 建筑限高
- 退线要求
- 配建要求
- 现状建筑量
- 规划建筑量
- 居住人口估算
- 就业人口估算
- 更新方式：保留、整治、改造、拆除重建、综合提升
- 所属规划单元和街坊

示例：

```json
{
  "id": "parcel_01",
  "type": "Parcel",
  "landUse": {
    "code": "0701",
    "name": "城镇住宅用地",
    "sourceStandard": "国土空间调查、规划、用途管制用地用海分类指南"
  },
  "areaSqm": 18500,
  "controls": {
    "farMax": 4.2,
    "buildingCoverageMax": 0.35,
    "greenRatioMin": 0.30,
    "heightMaxM": 100
  },
  "scenarioValues": {
    "scenario_a": {
      "far": 3.8,
      "residentialGfaSqm": 61000,
      "publicServiceGfaSqm": 1200,
      "estimatedResidents": 1850
    }
  }
}
```

### 5.5 RoadSegment 道路段

道路不是线，而是有等级、宽度、断面、交通功能和边界条件的对象。

关键字段：

- 道路名称
- 道路等级
- 红线宽度
- 车行道数量
- 非机动车道
- 人行道
- 公交设施
- 慢行连续性
- 是否适合开设机动车出入口
- 与地块边界关系

### 5.6 Entrance 出入口

出入口是规划判断中的高价值对象，传统 CAD 常常弱表达。

关键字段：

- 类型：机动车、人行、消防、货运、地下车库
- 服务对象：哪个地块或建筑
- 所在道路
- 距离交叉口距离
- 是否位于主干路
- 是否冲突公交站、人行横道、学校出入口、消防通道
- 推荐等级：可行、需优化、不建议、禁止

### 5.7 Facility 公共服务设施

公共服务设施是社区更新和完整社区判断的核心。

关键字段：

- 类型：幼儿园、小学、社区卫生服务、养老、文化、体育、便民商业、社区服务中心
- 服务半径
- 服务人口
- 建筑面积
- 用地面积
- 是否独立占地
- 是否可混合设置
- 所属服务圈层：5 分钟、10 分钟、15 分钟
- 缺口或冗余

### 5.8 BuildingMass 建筑体量

早期不需要精细 BIM，但需要规划级体量。

关键字段：

- 建筑用途
- 基底面积
- 建筑面积
- 高度
- 层数
- 退线
- 与道路、相邻地块、消防通道的关系
- 日照和视线分析入口

### 5.9 ConstraintOverlay 约束覆盖层

约束覆盖层用于表达不能简单变成地块属性的外部限制。

类型包括：

- 城镇开发边界
- 生态保护红线
- 永久基本农田
- 历史文化保护线
- 水体蓝线
- 绿线
- 黄线
- 紫线
- 洪涝风险区
- 地质灾害风险区
- 噪声影响区
- 轨道交通保护区

### 5.10 EvidenceSource 证据来源

每条外部数据都应有来源记录。

关键字段：

- 来源名称
- 来源类型：政府公开、标准规范、POI、OSM、遥感、实地调研、用户输入、模型推断
- 获取时间
- 数据许可
- 精度等级
- 可信度等级
- 适用范围
- 关联对象

## 6. 规则系统

### 6.1 规则分层

规则不要混在一起，应分成六类。

| 层级 | 类型 | 例子 | 软件处理 |
|---|---|---|---|
| L1 | 强制性底线 | 安全、防火、开发边界、保护线 | 红色错误，不应通过 |
| L2 | 法定控规控制 | 用地性质、容积率、限高、绿地率 | 红色或橙色错误 |
| L3 | 地方技术规定 | 深圳、上海、广州等地方参数 | 按城市和版本切换 |
| L4 | 专项规划约束 | 交通、市政、防灾、历史保护 | 作为叠加规则 |
| L5 | 设计导则建议 | 街道界面、慢行友好、混合功能 | 作为优化建议 |
| L6 | 软件推断评价 | 更新潜力、设施缺口、交通压力 | 标明为模型结果 |

### 6.2 规则对象

每条规则本身也应是数据。

```json
{
  "id": "rule_parcel_far_max",
  "name": "地块容积率不得超过控制值",
  "level": "L2",
  "severity": "error",
  "jurisdiction": "CN-GD-SZ",
  "effectiveFrom": "2025-01-01",
  "source": {
    "title": "深圳市城市规划标准与准则",
    "version": "2025修订汇总版",
    "chapter": "密度分区与容积率"
  },
  "targetObjectType": "Parcel",
  "condition": "parcel.controls.farMax != null",
  "expression": "parcel.scenarioValues[scenarioId].far <= parcel.controls.farMax",
  "message": "规划容积率超过地块控制容积率"
}
```

### 6.3 检查结果

规则检查结果也必须结构化。

```json
{
  "id": "check_0001",
  "ruleId": "rule_parcel_far_max",
  "objectId": "parcel_01",
  "scenarioId": "scenario_a",
  "status": "failed",
  "severity": "error",
  "observedValue": 4.6,
  "requiredValue": "<= 4.2",
  "message": "parcel_01 的规划容积率为 4.6，超过控制值 4.2。",
  "sourceTrace": [
    "rulesets/shenzhen_2025.json",
    "objects/parcels.geojson"
  ]
}
```

## 7. 智能推演能力

### 7.1 指标自动计算

基础计算：

- 地块面积
- 建筑基底面积
- 总建筑面积
- 计容建筑面积
- 容积率
- 建筑密度
- 绿地率
- 道路面积
- 公共开放空间面积
- 人均公共绿地
- 人均公共服务面积

### 7.2 人口自动估算

软件应支持多种人口估算模式：

- 按住宅建筑面积估算
- 按户数估算
- 按现状小区人口数据估算
- 按手机信令或热力数据估算
- 按夜间灯光和建筑量粗估
- 用户手动校正

关键是标明方法和置信度。

示例：

```json
{
  "estimatedResidents": {
    "value": 1850,
    "method": "residential_gfa_per_capita",
    "parameters": {
      "residentialGfaSqm": 61000,
      "sqmPerResident": 33
    },
    "confidence": "medium",
    "isUserOverridden": false
  }
}
```

### 7.3 公共服务设施推演

输入：

- 服务人口
- 年龄结构估计
- 设施现状
- 服务半径
- 地方配置标准

输出：

- 缺什么设施
- 缺多少规模
- 推荐布置在哪个地块
- 是否可混合设置
- 是否超过合理步行距离

例如：

- 居住人口增加后，自动检查幼儿园、小学、养老、社区卫生、文化体育、便民商业是否不足
- 自动生成 5/10/15 分钟服务圈覆盖图
- 自动比较“新增设施独立占地”和“嵌入底层公共服务空间”两种方案

### 7.4 道路和出入口推演

软件可以根据开发量和道路条件给出建议，但不应假装替代交通专项。

可做：

- 根据地块开发量估算出入口需求等级
- 判断出入口是否靠近交叉口
- 判断是否开在主干路上
- 判断是否影响公交站、人行横道、学校、医院等敏感点
- 推荐更合适的出入口边界
- 对道路断面进行基本完整性检查

不可过度承诺：

- 不应在早期宣称能精确模拟交通流
- 不应直接替代交通影响评价
- 不应自动决定道路红线宽度，只能做建议和风险提示

### 7.5 城市更新模式判断

软件可以把更新判断拆成多维度：

- 建筑老旧程度
- 土地利用低效程度
- 公共服务缺口
- 交通可达性
- 市政承载风险
- 历史文化价值
- 社区稳定性
- 实施难度

输出不应只是一个总分，而应是：

- 推荐更新模式
- 推荐理由
- 证据列表
- 风险提示
- 缺失数据

例如：

```text
parcel_03 推荐：综合整治优先，不建议直接拆除重建。
理由：
1. 建筑强度并不低，拆建增益有限。
2. 周边公共服务缺口主要在养老和社区活动空间，可通过存量嵌入改善。
3. 地块靠近历史风貌控制区，存在风貌管控风险。
4. 现状路网较窄，大幅增容可能带来交通压力。
```

## 8. 交互方式

### 8.1 主界面

主界面可以由五个区域组成：

- 中央：规划画布，支持地图、地块、道路、建筑体量、设施、控制线
- 左侧：项目树、图层、方案版本
- 右侧：对象属性、规则检查、智能建议
- 底部：指标表、问题列表、日志、对比面板
- 顶部：绘制、导入、推演、审查、导出

### 8.2 基本工作流

```mermaid
flowchart LR
  A["新建项目"] --> B["选择城市和规则版本"]
  B --> C["导入或绘制规划范围"]
  C --> D["生成规划单元、街坊、地块"]
  D --> E["设置用地和开发指标"]
  E --> F["导入证据数据"]
  F --> G["运行人口、设施、交通、更新推演"]
  G --> H["规则检查和风险提示"]
  H --> I["调整方案"]
  I --> J["多方案对比"]
  J --> K["导出图则、指标表和报告"]
```

### 8.3 对象编辑体验

用户点击地块，右侧不是普通属性表，而是规划语义面板：

```text
地块 parcel_01
用地：城镇住宅用地
面积：18,500 平方米
容积率：3.8 / 控制值 4.2
建筑密度：31% / 控制值 35%
绿地率：28% / 控制值 30%
估算人口：1,850 人
公共服务：养老设施不足，社区活动空间不足
交通：机动车出入口建议避开东侧主干路
风险：绿地率低于控制值
建议：降低基底、增加屋顶和公共开放空间不应直接计入法定绿地，需按地方规则校核
```

## 9. AI 应该怎么用

AI 不应替代规则引擎，也不应直接胡乱生成规划结论。

AI 更适合做：

- 把规范文本辅助转成规则草案
- 解释规则检查结果
- 生成报告初稿
- 根据问题清单提出方案调整方向
- 帮用户查询“为什么这里不合规”
- 做自然语言检索：例如“找出所有公共服务不足且适合更新的地块”

AI 不适合单独做：

- 法定合规判断
- 数值审查最终结论
- 没有依据的道路宽度决定
- 没有数据的拆除重建建议

正确架构是：

```text
用户操作
  -> 规划对象模型
  -> 确定性计算
  -> 规则引擎
  -> AI 解释和建议
  -> 用户确认
```

而不是：

```text
用户提问
  -> AI 直接给结论
```

## 10. 第一版 MVP

不要一开始做全功能城市规划软件。第一版应小而硬。

建议 MVP 主题：

> 面向城市更新片区的语义化地块规划与规则检查原型。

### 10.1 MVP 支持对象

第一版只支持：

- 项目边界
- 规划单元
- 街坊
- 地块
- 道路
- 出入口
- 公共服务设施
- 绿地和开放空间
- 控制线

暂时不做：

- 精细建筑单体建模
- 复杂交通仿真
- 地下空间
- 完整市政管网
- 真实审批流

### 10.2 MVP 支持功能

必须有：

- 新建项目
- 画地块、道路、设施
- 设置地块用地性质和开发指标
- 自动计算面积、容积率、建筑密度、绿地率
- 按住宅建筑量估算人口
- 按人口检查公共服务设施缺口
- 检查道路出入口基础风险
- 运行规则检查
- 生成问题清单
- 生成方案摘要报告
- 保存为 `.upf`

可选有：

- 导入 GeoJSON
- 导入 CAD/DXF
- 导出 GeoJSON
- 导出 PDF 报告
- 多方案对比

### 10.3 第一版规则范围

第一版规则不要贪多，可以先做四类：

- 地块指标规则：容积率、建筑密度、绿地率、限高
- 公共服务规则：按服务人口判断设施缺口
- 道路出入口规则：距离交叉口、道路等级、敏感点冲突
- 城市更新规则：更新模式建议和证据清单

### 10.4 第一版不要碰的高风险区

暂时不要承诺：

- 自动完成法定控规编制
- 自动通过政府审查
- 自动替代规划师
- 自动精准预测人流车流
- 自动判断所有国家和地方规范

第一版定位应是：

> 面向规划师和研究者的方案推演与规则辅助检查工具。

这个定位更专业，也更容易落地。

## 11. 技术架构建议

### 11.1 前端

适合用：

- TypeScript
- React
- MapLibre GL 或 OpenLayers
- Turf.js 做空间计算
- JSTS 做几何拓扑
- Zustand 或 Redux 做状态
- Monaco Editor 做规则编辑器

### 11.2 后端

原型阶段可以先本地化：

- Node.js 或 Python FastAPI
- SQLite / DuckDB
- GeoJSON / FlatGeobuf
- 本地规则 JSON

如果后续做平台：

- PostGIS 存空间对象
- 对象版本管理
- 规则服务
- 报告生成服务
- AI 服务

### 11.3 规则引擎

第一版不需要复杂 DSL，可以先用 JSON 规则 + TypeScript/Python 函数。

规则应有：

- id
- 名称
- 适用城市
- 适用对象
- 生效时间
- 来源
- 判断函数
- 严重等级
- 输出消息模板

### 11.4 几何和指标计算

常见计算：

- 多边形面积
- 缓冲区
- 最近距离
- 包含/相交
- 道路邻接
- 服务半径覆盖
- 地块合并拆分
- 可达性简化分析

### 11.5 未来可以接入的能力

- 遥感影像识别建筑和绿地
- OSM/高德 POI 辅助设施识别
- 街景图像识别街道品质
- 城市更新政策文本检索
- 大模型辅助规范条文转规则
- 多目标优化生成备选方案

## 12. 与 CAD/GIS/BIM 的关系

### 12.1 CAD

CAD 是导入和导出，不是内部核心。

支持：

- 导入 DXF/DWG 中的地块边界、道路、文字标注
- 将图层映射成规划对象
- 导出用于汇报或设计院协作的 CAD 图纸

但内部存储仍然是 UPF。

### 12.2 GIS

GIS 是空间底座。

支持：

- 坐标系统
- 地理底图
- 行政边界
- 道路路网
- POI
- 服务半径
- 空间叠加

但 GIS 不负责规划规则语义，UPF 负责。

### 12.3 BIM

BIM 可用于后期建筑深化。

规划阶段只需要建筑体量和关键指标，不需要一开始做门窗、构件、机电级 BIM。

## 13. 软件最有价值的地方

这个软件真正的价值不是“画图更方便”，而是：

- 把规划对象结构化
- 把规范规则机器化
- 把推断过程透明化
- 把多方案比较常态化
- 把城市更新从经验判断变成证据链判断
- 把图纸成果变成可计算、可审查、可追踪的数据资产

这比单纯做 CAD 插件更有学术价值，也更像一个可以长期发展的产品。

## 14. 可能的论文题目方向

可以考虑：

- 面向城市更新的语义化规划数据模型与辅助审查方法研究
- 基于规则引擎的控制性详细规划方案合规性检查系统设计
- 面向存量更新的规划对象模型与多源证据推演平台研究
- 城市更新片区公共服务设施缺口识别与规划方案推演工具研究
- 从 CAD 图纸到语义化规划模型的转换方法与应用研究

## 15. 下一步工作

下一步不应该马上写完整软件，而应先做三件事：

1. 定义 UPF 0.1 的最小对象集
2. 定义 20 条左右可执行规则
3. 做一个可交互原型：画地块、设指标、跑检查、出报告

建议第一版原型目标：

> 用户画出一个城市更新片区，设置若干地块的用地和指标，软件自动计算人口、公共服务缺口、基础交通风险和规范问题，并输出一份结构化规划诊断报告。

这就已经足够像一个新的东西，而不是 CAD 的附属品。
